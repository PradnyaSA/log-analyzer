# Agent Spec

## Overview
An ambient AI agent that monitors live ADK audit logs streamed via Pub/Sub. When an anomaly threshold is crossed (e.g., a spike in 404 errors), the agent self-triggers, runs a grounded RCA pipeline, and produces a fully cited report — every finding tied to an exact source span in the log corpus. The agent pauses for HITL approval before filing a Jira ticket and transitioning the incident status.

Deployable as an embeddable component in a multi-agent environment (MCP surface) or as a standalone ambient agent on Agent Runtime.

## Example Use Cases

- **Anomaly-triggered RCA:** Pub/Sub delivers an event when 404 errors on the concierge agent exceed a 20% spike threshold. Log-Analyzer wakes, reads the relevant audit log window, identifies the Top-K cutoff as root cause, writes an RCA report, and pauses for human approval to file a Jira story.
- **Manual triage:** An engineer invokes the agent via CLI with temporary read-only access to a log file; the agent returns a structured RCA report with line-level citations.

## Tools Required

| Tool | Purpose | Auth |
|------|---------|------|
| Pub/Sub subscription reader | Consume incoming anomaly-triggered events from the audit log stream | GCP service account |
| Log reader (GCS or local) | Read bounded log/audit file windows for analysis | GCP service account (read-only) |
| Jira ticket creator | File a Jira story on HITL approval | Jira API token (Secret Manager) |
| Incident status updater | Transition incident to triage-completed | Jira/incident API token (Secret Manager) |

## Constraints & Safety Rules

- Log access is **read-only** — the agent must never write to or delete log files.
- All external write actions (Jira ticket, incident status change) require **explicit HITL approval** before execution.
- Every claim in the RCA report must be traceable to an **exact log line** (no hallucinated citations).
- The agent must **not** build a general-purpose alerting or notification pipeline — its only write-back actions are: one incident-status transition and one Jira ticket.
- No petabyte-scale log ingestion — designed for bounded sample log sets, not production-volume telemetry.

## Success Criteria

- Anomaly event received via Pub/Sub → RCA report generated with ≥ 1 exact log-line citation per finding.
- HITL approval flow: agent pauses correctly, resumes on approval, files Jira ticket and updates incident status.
- False-positive rate: < 10% of triggered analyses produce incorrect root cause attribution (validated via eval).
- End-to-end latency from Pub/Sub event to RCA report ready for human review: < 5 minutes on bounded log sets.

## Architecture Notes

- **Deployment target:** Agent Runtime (managed by Google; sessions handled automatically)
- **Trigger:** Pub/Sub event (anomaly threshold crossed on audit log stream)
- **Session type:** Not set (Agent Runtime manages sessions internally)
- **Template:** `adk` (standard ADK agent; A2A protocol built in)
